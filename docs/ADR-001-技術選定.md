## コンテキスト

### 背景

イベント（2027-01-31）でゲストが撮った写真を1箇所に集めるWebアプリを開発する。
個人利用のため商用スケールは不要。50〜100人のゲストが同時アクセスする想定。
UI/UXを作り込んでゲストが迷わず使えることを最優先する。

### 課題

- ゲストがアプリインストールなしで使えること（ブラウザ完結）
- 写真アップロードのパフォーマンス（待たせない）
- 低コスト運用（個人利用のため）
- UI/UXの作り込みやすさ
- 50〜100人の同時アクセスでのデータ整合性

### 制約

- 開発者1人（自分のみ）
- git pushデプロイが望ましい
- Docker不要が望ましい

---

## 検討した選択肢

### 選択肢1: Amplify + S3のみ（DB不要）

**概要**: S3にJSONサイドカーでメタデータ管理。manifest.jsonで一覧取得。

| 評価軸 | 評価 |
|-------|------|
| サービス数 | 2（Amplify + S3） |
| 同時書き込み安全性 | **低**（manifest.json競合リスク） |
| データ整合性 | **低**（JSON手動管理） |
| フィルタ・ソート | クライアント側JSで処理 |
| 開発体験 | S3 SDK直叩き |

**デメリット:**
- manifest.jsonの同時書き込みでデータ消失リスク
- ニックネーム重複チェックが結果整合性（稀に漏れる）
- フィルタ・ソートがクライアント依存

---

### 選択肢2: Amplify + S3 + Turso + Prisma（採用）

**概要**: メタデータをTurso（ホスト型SQLite）+ Prismaで管理。S3は写真ファイルのみ。

| 評価軸 | 評価 |
|-------|------|
| サービス数 | 3（Amplify + S3 + Turso） |
| 同時書き込み安全性 | **高**（DB管理） |
| データ整合性 | **高**（UNIQUE制約、トランザクション） |
| フィルタ・ソート | **SQLクエリで一発** |
| 開発体験 | **Prismaの型安全なクエリ** |

**メリット:**
- UNIQUE制約でニックネーム重複チェックが確実
- SQLクエリでフィルタ・ソートが容易（WHERE scene='懇親会'）
- Prismaの型安全なクエリで開発が楽
- HTTP接続でサーバーレス（Lambda）対応、コールドスタートなし
- 無料枠が十分（9GBストレージ、10億行読み/月）

**デメリット:**
- サービス数が1つ増える（Turso）
- Prisma + libSQLアダプターの初期設定が必要

---

### 選択肢3: Cloudflare Pages + D1 + R2

**概要**: Cloudflareエコシステムで統一。D1（SQLite）でデータ管理、R2で写真。

| 評価軸 | 評価 |
|-------|------|
| サービス数 | 1（Cloudflare内で完結） |
| 転送量 | **完全無料** |
| Next.js対応 | 中（Amplifyほど成熟していない） |
| Prisma対応 | Yes（@prisma/adapter-d1） |

**デメリット:**
- Cloudflareプラットフォームにロックイン
- Next.js App Routerの対応がAmplifyほど成熟していない

---

## 決定

### 採用する選択肢

**選択肢2: Amplify + S3 + Turso + Prisma** を採用する

### 決定理由

1. 50〜100人の同時アクセスでデータ整合性が確実（UNIQUE制約、トランザクション）
2. Prismaの型安全なクエリで開発効率が高い
3. Turso無料枠が十分（9GBストレージ）、コールドスタートなし
4. Next.js App RouterのAmplify対応が最も成熟

### トレードオフ

| 得られるもの | 失うもの |
|-------------|---------|
| データ整合性・同時書き込み安全性 | サービス数最小（2→3に増加） |
| Prismaの型安全なクエリ | S3のみのシンプルさ |
| SQLによるフィルタ・ソート | Cloudflareの転送量無料 |
